01The Promise Is Real. The Plumbing Is a Nightmare.

Every CTO has seen the demo. Three lines of LangChain, a vector store, a smiling chatbot that actually answers the question. The room nods. The pilot ships. And then somebody asks the question that quietly torpedoes the project: how do we run this in production?

The honest answer, in most enterprises, is we built a one-off. A Python script on an EC2 box, a Pinecone key in a Slack DM, a model API call wrapped in retry logic written by an intern who has since left. The promise of agentic apps is real — but the plumbing underneath them looks like a 2014 Hackathon project that nobody's taken responsibility for.

02What an Agentic App Actually Needs From Its Platform

Set aside the model for a moment. The model is the easy part — somebody else makes it, you pay for tokens, you call an API. What's hard is everything around it. An agentic app is not a model; it's a process that calls models, calls tools, holds state, reads from your data, writes to your databases, and runs for as long as the user needs an answer.

That process needs the same things any other long-running enterprise workload needs — and a few new ones besides. It needs identity. It needs network policy. It needs secrets that don't live in dotenv files. It needs observability for prompts and tool calls, not just CPU graphs. And it needs boring, predictable deployment.

03Spring AI Gives You the Blueprint

Here's the part that surprised me. If you've already invested in Spring — and most large engineering shops have, whether they admit it or not — Spring AI gets you most of the way there without inventing a new framework. The abstractions are right where you'd expect them: ChatClient, VectorStore, FunctionCallback. The same dependency injection you use for a controller now wires up a model call.
